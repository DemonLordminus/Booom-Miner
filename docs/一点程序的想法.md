# 程序设计思想
其实如果能够做到各自实现各自的模块，我们完全不需要使用一套相同的代码风格。大伙各自做好各自的模块，拼起来能用就行。
不过我个人还是编写一点自己的经验之谈，仅作参考。
## 组合优于继承
使用继承往往会带来更加复杂的结构，如果子类中出现了例外，就要去修改父类。而父类的修改又会导致所有子类都发生改变。
因此个人提倡，除了类似单例模式和Unity的MonoBehavior这种固定的继承关系，不要自己创造新的继承关系。
一切继承都尽可能使用组合来代替，存储共有数据使用变量成员类，调用共有数据、说明共有方法、使用多态都使用接口Interface。
在原理上是可以使用组合完全代替继承的，不过相对应的，所需要写的代码会变多，至少你需要给每个接口都实现一遍对应的方法，虽然大多数情况下直接复制即可。
## 不要过早优化
在内容做完之前，都不考虑优化问题，相信新时代的CPU。通常来说，一个月的项目时间支撑不起优化环节，并且2DUnity游戏往往不会有到严重影响性能的部分。
因此，在项目的开发中，都优先考虑代码的易读性以及编写效率，而不要去考虑优化问题。
## 组件式开发思路
这里也和组合有关系，简单来说，就是每一个功能对应一个类，尽可能将功能分散开。然后再由一个本体脚本，将这些类组合起来。
这些组件式的类可以继承Unity的MonoBehavior，直接做成类似Unity插件的感觉。也可以不继承MonoBehavior，加上序列化后作为成员变量使用。

# 系统模块拆解测试
接下来我将按照个人理解，初步对所提供的功能拆解成各个代码模块。非单例的各模块之间的代码尽可能不会有直接联系，具有依赖关系且都在同一物体上的模块允许直接产生单向联系。
如果各模块之间需要产生联系，则通过单例模式以及发布订阅模式，创建一个具有多态的对应的事件类，并由一个事件处理中心来负责代码传递。
* 输入模块
	* 采用Unity的new input system与单例模式，创建一个Input Manager，一切的输入由Input Manager接收并处理，其他模块可以通过订阅相关的input事件或者访问变量来获取输入值。
* 自制物理刚体插件
	* 由于Unity内置的Rigid Body 2D组件所带的重力、碰撞等，只能实现对现实中物理的模拟。因此我个人认为，需要自己编写一个简单的处理插件，使用RigidBody2D的MovePosition函数，自己实现重力、碰撞力、加速度等功能。
	* 该插件可能包含的功能：
		* 提供一个移动的函数接口，由其他脚本提供移动的方向。
		* 提供一个地面检测，使用Physical2D.OverlapCircle即可。
		* 开放加速度、最大速度、重力、最大下坠速度等参数。
